# Localized Web Components

This article extends [Web Components](:Zolinga Core:Web Components) with translation-specific behavior. Read the base article first for registration, `loadContent()`, readiness, and general architecture.

## When to Use `web-component-intl.js`

If a component loads an HTML template that contains translatable strings, import from `/dist/zolinga-intl/js/web-component-intl.js` instead of the base `/dist/system/js/web-component.js`:

```javascript
import WebComponent from '/dist/zolinga-intl/js/web-component-intl.js';

export default class MyWidget extends WebComponent {
    constructor() {
        super();
        this.ready(this.#init());
    }

    async #init() {
        this.#root = await this.loadContent(
            import.meta.url.replace('.js', '.html'),
            { mode: 'open' }
        );
    }
}
```

`WebComponentIntl` extends the base class and overrides `rewriteURL()` so that `loadContent()` automatically loads a locale-specific template when the page language is not `en-US`.

## Locale-Aware Template Loading

When `document.documentElement.lang` is something other than `en-US` (for example `cs-CZ`), `loadContent('my-widget.html')` rewrites the URL before fetching:

| Requested file | `en-US` | `cs-CZ` | `de-DE` |
|----------------|---------|---------|---------|
| `my-widget.html` | `my-widget.html` | `my-widget.cs-CZ.html` | `my-widget.de-DE.html` |
| `my-widget.css` | `my-widget.css` | `my-widget.cs-CZ.css` | `my-widget.de-DE.css` |

The locale code is inserted **just before the file extension**. If the localized file does not exist on the server, the request falls back to the base file.

## Marking Strings in the HTML Template

Inside the `.html` file loaded by `loadContent()`, mark translatable content exactly as you would in any static HTML page:

```html
<!DOCTYPE html>
<html>
  <head>
    <meta name="gettext" content="translate"/>
    <link rel="stylesheet" href="my-widget.css"/>
  </head>
  <body>
    <h1 gettext=".">Hello, World!</h1>
    <button gettext="." title="Submit form">Send</button>
  </body>
</html>
```

Requirements:
- Add `<meta name="gettext" content="translate"/>` in `<head>` so the extractor recognizes the file.
- Use `gettext="."` for text content, `gettext="alt"` for images, `gettext="title"` for titles, etc.
- Use `gettext-context` to disambiguate short or ambiguous strings.
- Add `<!-- TRANSLATORS: ... -->` comments for context.

See [HTML Translations](:Zolinga Core:HTML Translations) for the full attribute syntax, nested-element rules, and translator-comment placement.

## Marking Strings in JavaScript

For dynamic strings emitted from the `.js` file, import the domain-bound helper:

```javascript
import { gettext, ngettext } from '/dist/zolinga-intl/gettext.js?my-module';

this.broadcast('message', {
    type: 'success',
    message: gettext('Settings saved successfully.')
});
```

The `?my-module` query string sets the gettext domain. Always use literal strings inside `gettext()` — never variables — so the extractor can find them at parse time.

See [JavaScript Translations](:Zolinga Core:JavaScript Translations) for the full API, plural forms, and translator-comment syntax.

## When You Do NOT Need `web-component-intl.js`

Use the base `/dist/system/js/web-component.js` when:
- The component has **no HTML template** (pure JS component).
- The template contains **no translatable strings** (only icons, charts, or purely dynamic data).
- The component is **never localized** (internal admin tools, debug components).

## File Layout

```
modules/my-module/install/dist/web-components/my-widget/
  my-widget.js          # imports web-component-intl.js
  my-widget.html        # base template with gettext attributes
  my-widget.css         # base styles
  my-widget.md          # component documentation
  my-widget.cs-CZ.html  # generated by gettext:compile
  my-widget.de-DE.html  # generated by gettext:compile
```

The locale-specific `.html` and `.css` files are **generated automatically** by `bin/zolinga gettext:compile`. Do not create them manually.

## Workflow

1. **Mark** strings in `.html` with `gettext` attributes and in `.js` with `gettext()` calls.
2. **Extract** strings: `bin/zolinga gettext:extract --domains=my-module`
3. **Translate** `.po` files in `modules/my-module/locale/`.
4. **Compile** translations: `bin/zolinga gettext:compile --domains=my-module`
5. **Verify** — switch the page locale and confirm the component loads the localized template.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Using `web-component.js` for a component with translatable HTML | Switch to `web-component-intl.js` |
| Forgetting `<meta name="gettext" content="translate"/>` in the template | Add it so the extractor recognizes the file |
| Manually creating `my-widget.cs-CZ.html` | Let `gettext:compile` generate it from `.po` |
| Hard-coding English strings in JS without `gettext()` | Import `gettext.js` and wrap strings |
| Using variables inside `gettext()` calls | Use literal strings only; substitute variables at runtime with `sprintf` patterns |
